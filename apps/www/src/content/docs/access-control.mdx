Cliparr can remember provider connections, but it does not have a full user system with app-level accounts, roles, or permissions. Treat access to the Cliparr web app as access to the connected sources and active media sessions.

<Callout title="Provider login is not Cliparr access control" tone="warning">
  Plex and Jellyfin authentication lets Cliparr connect to those sources. It
  does not limit who can open the Cliparr app once they can reach it.
</Callout>

## Recommended deployment

Keep Cliparr on `localhost`, a trusted private network, a VPN, or behind an authenticated reverse proxy. If you expose Cliparr outside a trusted network, put authentication in front of it with the proxy or access gateway you already trust.

Common options include:

- Caddy, Nginx, or Traefik with an external auth service.
- Authentik, Authelia, or another identity-aware reverse proxy.
- Cloudflare Access, Tailscale, WireGuard, or another private-access layer.

## Reverse proxy basics

When Cliparr is behind a reverse proxy, preserve the `Host` header and pass `X-Forwarded-Proto`. See [Configuration](/docs/configuration) for the core reverse proxy and HTTPS requirements.

Use HTTPS for remote access. The editor relies on browser WebCodecs, and supporting browsers require a secure context outside `localhost` or `127.0.0.1`.

## Caddy basic auth example

Caddy can put a simple username and password in front of Cliparr with `basic_auth`. Use this only over HTTPS, and make sure users cannot bypass Caddy by reaching Cliparr's container port directly.

First, generate a hashed password:

```bash
docker run --rm -it caddy:2 caddy hash-password
```

Then use the generated hash in a `Caddyfile`:

```caddyfile
cliparr.example.com {
    encode zstd gzip

    basic_auth {
        cliparr_user REPLACE_WITH_CADDY_HASH
    }

    reverse_proxy cliparr:3000 {
        header_up Host {host}
        header_up X-Forwarded-Proto {scheme}
    }
}
```

If Cliparr and Caddy run in the same Docker Compose project, expose Cliparr only to the Compose network and publish ports from Caddy:

```yaml
services:
  cliparr:
    image: ghcr.io/techsquidtv/cliparr:latest
    environment:
      - APP_KEY=replace-this-with-a-32-character-secure-random-string
    expose:
      - "3000"
    volumes:
      - cliparr-data:/data
    restart: unless-stopped

  caddy:
    image: caddy:2
    depends_on:
      - cliparr
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy-data:/data
      - caddy-config:/config
    restart: unless-stopped

volumes:
  cliparr-data:
  caddy-data:
  caddy-config:
```

If you run Cliparr outside Docker Compose, bind Cliparr to localhost and proxy to that local address:

```bash
docker run -d \
  --name cliparr \
  -p 127.0.0.1:3000:3000 \
  -e APP_KEY="your-32-char-stable-random-secret" \
  -v cliparr-data:/data \
  ghcr.io/techsquidtv/cliparr:latest
```

Then use `reverse_proxy 127.0.0.1:3000` in the Caddyfile instead of `reverse_proxy cliparr:3000`.

## What to protect

Protect every Cliparr route with the same access policy, including:

- The web app.
- `/api/*` routes.
- Provider connection and callback routes.
- Media proxy routes used while previewing and exporting clips.

## Credential storage

Provider credentials and remembered provider sessions are stored under the Cliparr data directory and encrypted or signed with `APP_KEY`. Keep `APP_KEY` stable, private, and backed up with the matching data directory.
